## Findings

### Finding 1 â€” No validation guard for NaN after parseInt

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| Class | Missing input validation |
| File | `src/controllers/userController.js` |
| Line | 19 |
| Description | `parseInt(req.params.id, 10)` silently returns `NaN` for any non-numeric string (e.g., `abc`, `../etc`, `1e9`). The downstream `users.find(u => u.id === userId)` comparison with `NaN` always returns `false`, so the request falls through to the existing 404 branch â€” which is the correct observable behaviour today. However, there is no explicit guard that rejects `NaN` with a `400 Bad Request`. This means: (1) Any caller that relies on distinguishing a malformed request (400) from a missing resource (404) gets an incorrect status code, which can lead to security logic in upstream proxies or API gateways making incorrect routing/rate-limiting decisions. (2) If the data-access layer is ever replaced with a real database query, passing `NaN` to an ORM or raw query builder without a guard may trigger unexpected behaviour or injection vectors depending on the library. |
| Remediation | Add an explicit NaN check immediately after the `parseInt` call: if `isNaN(userId)` return `res.status(400).json({ error: 'Invalid user ID' })`. This makes the invalid-input case explicit, returns the semantically correct HTTP status, and prevents `NaN` from propagating to any future data-access layer. |

### Finding 3 â€” Full user list endpoint exposes email addresses without pagination or filtering

| Field | Value |
|-------|-------|
| Severity | LOW |
| Class | Sensitive data exposure |
| File | `src/controllers/userController.js` |
| Line | 37â€“39 |
| Description | `getAllUsers` returns the entire in-memory `users` array, including email addresses, in a single response with no pagination, filtering, or field-level access control. This is a data-minimisation violation: callers receive more data than is typically needed for a list view, and bulk enumeration of PII (emails) is trivially easy for any caller with network access. |

### Finding 4 â€” Express version uses a caret range, allowing unvetted minor/patch upgrades

| Field | Value |
|-------|-------|
| Severity | INFO |
| Class | Unsafe dependencies |
| File | `package.json` |
| Line | 15 |
| Description | The Express dependency is declared as `"^4.18.2"`, which permits npm to install any `4.x.y` release newer than `4.18.2` without a lock-file review. If `package-lock.json` is regenerated or deleted, a future `npm install` could pull in a minor or patch version that introduces a regression or a yet-to-be-disclosed CVE. While `^` ranges are common practice, for a production service the risk of unreviewed transitive dependency changes is non-trivial. |
| Remediation | Pin the dependency to an exact version (e.g., `"express": "4.18.2"`) and manage upgrades deliberately through a dependency-update workflow (e.g., Dependabot or Renovate with review gates). Commit `package-lock.json` and verify it is not in `.gitignore`. |
