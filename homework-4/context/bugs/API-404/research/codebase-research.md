# Codebase Research — API-404

> Generated by: Bug Researcher
> Date: 2026-02-24

---

## Bug Summary

`GET /api/users/:id` returns a 404 "User not found" response for every valid user ID because the controller compares the URL path parameter (always a `string`) against numeric IDs stored in the in-memory array using strict equality (`===`), which always evaluates to `false`.

---

## Root Cause

Express route parameters are always extracted as JavaScript **strings** (e.g., `"123"`). The in-memory `users` array stores IDs as JavaScript **numbers** (e.g., `123`). In `src/controllers/userController.js`, `Array.prototype.find` uses the strict equality operator (`===`) to compare `req.params.id` (a string) with each `user.id` (a number). Because `"123" === 123` is `false` in JavaScript, the lookup never finds a match regardless of which ID is requested, and the handler unconditionally returns a 404 response.

---

## Evidence

### File: `src/controllers/userController.js`
- **Line 8–10**: `{ id: 123, ... }, { id: 456, ... }, { id: 789, ... }`
  - **Issue**: All IDs in the mock database are stored as numeric literals (`number` type).
- **Line 19**: `const userId = req.params.id;`
  - **Issue**: `req.params.id` is always a `string` as produced by Express's URL parser. No type conversion is applied, so `userId` is `"123"` (string) when the caller requests `/api/users/123`.
- **Line 23**: `const user = users.find(u => u.id === userId);`
  - **Issue**: Strict equality (`===`) between `u.id` (number) and `userId` (string) is always `false`. The array comment on lines 21–22 explicitly acknowledges this: `// BUG: req.params.id returns a string, but users array uses numeric IDs // Strict equality (===) comparison will always fail: "123" !== 123`. The `find` call returns `undefined` for every possible input.

### File: `src/routes/users.js`
- **Line 14**: `router.get('/api/users/:id', userController.getUserById);`
  - **Issue**: The `:id` segment captured here is passed through to the controller without any coercion or middleware that would convert it to a number. This is the entry point where `req.params.id` first becomes a string.

### File: `server.js`
- **Line 16**: `app.use(userRoutes);`
  - **Note**: No global parameter-coercion middleware is registered before the routes, so no automatic conversion from string to number occurs at the server level.

## Data Flow

```
HTTP GET /api/users/123
  └─> server.js (line 16) — app.use(userRoutes)
        └─> src/routes/users.js (line 14) — router.get('/api/users/:id', ...)
              └─> Express URL parser extracts req.params.id = "123"  ← string
                    └─> src/controllers/userController.js (line 19)
                          const userId = req.params.id;    // "123" (string)
                          └─> (line 23)
                                users.find(u => u.id === userId)
                                  // u.id = 123 (number) === "123" (string)
                                  // false for every element → returns undefined
                          └─> (lines 25–27)
                                if (!user) → true → res.status(404).json({ error: 'User not found' })
```

The type mismatch occurs between line 19 (string extraction) and line 23 (strict numeric comparison).

## Recommended Fix Direction

The fix should be applied in `src/controllers/userController.js` at the point where `req.params.id` is read. Before performing the `users.find` lookup, the string value should be converted to a number (e.g., using `parseInt`, `Number()`, or the unary `+` operator), and the resulting parsed value should be validated to ensure it is a finite integer before the comparison. Alternatively, the comparison predicate on line 23 could be changed to use loose equality (`==`) or to coerce one side explicitly, though explicit numeric parsing is generally clearer and safer. No changes to `server.js` or `src/routes/users.js` are required.
