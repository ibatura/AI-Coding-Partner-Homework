## Bug Summary

`getUserById` in `src/controllers/userController.js` extracts `req.params.id` as a JavaScript string and then compares it against numeric IDs in the in-memory `users` array using strict equality (`===`). Because `"123" === 123` is always `false` in JavaScript, the `Array.prototype.find` lookup never matches any user, causing every `GET /api/users/:id` request to return a 404 "User not found" response regardless of whether the ID exists.

---

## Fix Strategy

Convert the string route parameter to an integer at the point of extraction (line 19) using `parseInt(req.params.id, 10)`. This is the minimal, self-contained change: it fixes the type mismatch at the earliest possible point in the data flow, keeps the existing strict-equality comparison on line 23 semantically correct (number vs. number), and does not require changes to the route definition or any middleware. Using the radix `10` with `parseInt` makes the base explicit and avoids unexpected octal/hex interpretations of pathological input. If the caller supplies a non-numeric ID string, `parseInt` returns `NaN`; the existing `find` predicate will then return `undefined`, and the existing 404 branch will handle it correctly — meaning no additional validation logic is required.

| Before | `  const userId = req.params.id;` |
| After | `  const userId = parseInt(req.params.id, 10);` |
| Rationale | `req.params.id` is always a string in Express 4.x. The `users` array stores IDs as numbers. Wrapping the extraction in `parseInt(..., 10)` coerces the value to a number before it reaches the strict-equality comparison on line 23, so `u.id === userId` becomes a number-to-number comparison that evaluates correctly. No other lines need to change. |
