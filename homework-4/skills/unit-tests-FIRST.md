# Unit Tests — FIRST Principles Skill

This skill defines the **FIRST** quality criteria that every unit test generated by the
**Unit Test Generator** agent must satisfy before it is written or committed.

---

## The Five FIRST Principles

### F — Fast

- Each test executes in **milliseconds**.
- No real I/O: no file reads/writes, no network calls, no database connections.
- All external dependencies (filesystem, HTTP clients, databases, clocks) are **mocked or stubbed**.

> Failure mode: a test that waits on a real HTTP response, sleeps, or queries a live DB is not Fast.

---

### I — Independent (Isolated)

- Each test sets up its **own fresh state** — no reliance on state left by a previous test.
- Tests can run in **any order** and still produce the same result.
- No shared mutable variables between test cases.

> Failure mode: a test that passes only when run after another test, or that modifies a module-level variable that other tests read, is not Independent.

---

### R — Repeatable

- The test produces the **same pass/fail result on every run**, in every environment.
- No dependency on real clocks (`Date.now()`, `time.Now()`), random number generators, environment variables with varying values, or machine-specific file paths.
- Seed random data or mock clocks when the code under test uses them.

> Failure mode: a test that passes on the developer's machine but fails in CI is not Repeatable.

---

### S — Self-Validating

- The test ends with **at least one meaningful assertion** (e.g., `expect`, `assert`, `assertEquals`).
- Pass/fail is **automatic** — no human needs to read logs or console output to determine the result.
- A test that prints output but makes no assertions is not self-validating.

> Failure mode: a test that runs without error but asserts nothing, relying on the absence of exceptions as proof of correctness, is not Self-validating.

---

### T — Timely

- Tests cover **only the code that was just changed**.
- Each test name **describes the specific scenario** being tested (happy path, edge case, regression).
- Do not add tests for unchanged, pre-existing code — that is out of scope.

> Failure mode: a test added "while we're here" for unrelated functionality is not Timely.

---

## FIRST Compliance Checklist

Before writing each test, verify all five items:

| # | Check | Pass condition |
|---|-------|----------------|
| 1 | Fast | No real I/O; all external deps mocked |
| 2 | Independent | Fresh state; no ordering dependency |
| 3 | Repeatable | Deterministic; no real clocks/random/env-paths |
| 4 | Self-validating | At least one `expect`/`assert` per test |
| 5 | Timely | Covers only changed code; name describes scenario |

All five must be **Yes** before the test is written.

---

## Required Reporting Format

The Unit Test Generator must include a FIRST compliance table in `test-report.md`:

```markdown
| Principle       | Satisfied | Notes                      |
|-----------------|-----------|----------------------------|
| Fast            | Yes / No  | {reason if No}             |
| Independent     | Yes / No  | {reason if No}             |
| Repeatable      | Yes / No  | {reason if No}             |
| Self-validating | Yes / No  | {reason if No}             |
| Timely          | Yes / No  | {reason if No}             |
```
